# 工艺编辑器培训与 CF 参考资料

## 1. 资料可信度说明

本文按以下优先级处理信息：

1. 真实设计方案的运行日志和输出 JSON；
2. 同业务域、同入口的成功脚本；
3. 工艺编辑器“可用类型/脚本”列表；
4. 培训视频中可见的界面、代码和操作过程；
5. CF 文档中的历史说明；
6. 未经运行验证的推断。

培训视频和 CF 文档用于理解平台能力，但不同时间、业务线和租户开放的类型可能不同。脚本实现应以当前环境实际显示的“可用类型/脚本”和运行证据为准。

## 2. 培训视频概要

培训视频标题为“工艺编辑器开发内容沟通”，录制时间为 2025-04-22。视频展示了从配置到设计工具验证的完整闭环：

```text
需求分析
  -> 定义自定义对象
  -> 配置业务数据
  -> 编写脚本
  -> 创建测试用例
  -> 保存并发布版本
  -> 在酷家乐 5.0 真实触发
  -> 检查模型、商品、费用和输出
```

视频中的主要示例包括：

- 前挡水与垫条关系配置；
- 根据台面尺寸计算垫条长度；
- 前挡水圆弧工艺费计算；
- 自定义对象和脚本模板复用；
- 测试用例变量值及返回结果查看；
- 发布后在设计工具中验证商品和参数。

视频传达的核心不是某个函数，而是“对象结构、配置数据、执行脚本、发布版本和真实场景”必须形成闭环。

## 3. 工艺编辑器的组成

### 3.1 自定义对象

自定义对象用于定义脚本可以读取、传递或返回的数据结构，类似 Java/TypeScript 的对象类型。

重点字段：

- 对象名称和描述；
- 业务模块和业务线；
- 自定义对象名称；
- 继承类型；
- 属性 key、类型、必填、数组和索引；
- 是否开启数据配置。

若一个自定义对象需要作为另一个对象的属性，历史 CF 资料建议继承 `parammodelMap/ParamModelInstance`，避免后端转换失败。最终仍需以当前“继承类型”选项和测试结果确认。

### 3.2 数据配置

自定义对象只定义结构，数据配置负责绑定真实业务数据，例如：

- 品牌商品；
- 输出编码；
- 挡水和垫条关联关系；
- 费用计算规则；
- 工具线和业务线；
- 作为查询条件的索引值。

常见读取方式：

```ts
let config = getOneConfig(
    ToolType.currentToolType(),
    ConfigObjectDef,
    indexValue
);
```

索引字段、业务线、商品授权或发布版本不正确时，即使脚本本身没有语法错误，也可能查不到配置。

### 3.3 自定义脚本

脚本具有唯一标识、名称、描述、业务模块、入参和返回值。唯一标识用于系统入口识别或其他脚本导入。

必须区分：

- **系统入口**：由平台在固定时机调用，函数名、入参和返回值通常固定；
- **子脚本**：只有被入口调用时才执行；
- **模板**：用于复用对象或脚本结构，不等于已发布的业务实现。

### 3.4 版本和发布

保存成功只说明编辑器接受了内容，不代表前台已经使用：

1. 自定义对象和依赖脚本先发布；
2. 入口脚本后发布；
3. 确认列表存在“当前版本”；
4. 在真实业务动作中重新触发；
5. 查询日志并检查最终输出。

## 4. 先选业务域，再选对象

不同业务域开放的对象和函数不同。不能因为某个域找不到一种类型，就推断整个工艺编辑器不支持该能力。

| 需求 | 常见业务域/入口 | 典型对象 |
| --- | --- | --- |
| 台面洞、挡水、垫条、工艺费、台面虚拟商品 | 台面 `generateFee` 或已开放的台面输出入口 | `Countertop`、`Fee` |
| 设计数据输出前统一增改模型 | 设计后处理 `postProcess` | `PostProcessDesignData`、`DesignPostProcessResult` |
| 模型告警和规则提示 | `ruleCheck` | `DesignData`、`ModelRuleResult` |
| 订单字段和提审计算 | 订单后处理 | `DcsOrder` |
| 轮廓、洞口、尺寸和加工数据 | 对应图纸入口 | 图纸、路径和几何对象 |

选择入口时回答四个问题：

1. 脚本在设计中、输出时还是订单阶段触发？
2. 需要修改原对象、返回费用，还是追加子模型？
3. 当前业务模块是否暴露目标类型？
4. 最终结果应该在画布、日志、订单字段还是输出 JSON 中出现？

## 5. 语法特点与限制

工艺编辑器使用自研的类 TypeScript 语法，不能按完整 TypeScript 或 JavaScript 理解。

常用写法：

```ts
let numbers: number[] = [1, 2, 3, 4];
let filtered = numbers.filter(value => value < 4);
let mapped = numbers.map(value => numberToString(value));
```

字符串日志优先使用数组 `join`：

```ts
printSystemLog([
    "工艺编辑器：1 数量",
    numberToString(count)
].join(" - "));
```

历史资料和实操中常见限制：

| 项目 | 注意事项 |
| --- | --- |
| 变量名 | 同一脚本尽量不要重复声明，包括不同分支 |
| 相等判断 | 使用 `===` 和 `!==` |
| 字符串 | 优先双引号，不用 `+` 拼接 |
| 数组 | 常用 `filter`、`map`、`foreach`、`push`、`size`、`distinct` |
| 不可用能力 | 常见环境不支持 `find`、`findIndex`、`a++`、`a--` |
| 异常处理 | 不使用 `try/catch` |
| 类型声明 | 不在脚本内定义 `interface` 或 `type` |
| 箭头函数 | 不标注箭头函数返回类型 |
| 对象和数组 | 最后一项后不要保留尾逗号 |
| 返回语句 | 分支中的提前 `return` 要用最小脚本验证执行行为 |
| 属性名 | 避免把 `type` 作为自定义对象属性 key |

这些限制可能随引擎升级变化。遇到“格式错误”时，应缩减为最小函数，再逐段恢复代码。

## 6. 测试、日志与 debugger

### 6.1 测试用例

编辑器测试适合验证：

- import 和类型是否可用；
- 函数签名和返回类型；
- 变量值和局部计算；
- 配置对象是否能查询；
- 最小输入是否能运行。

编辑器测试不能替代真实设计验证，因为系统入口、商品权限和输出链路可能只在真实流程中生效。

### 6.2 日志

```ts
import { printSystemLog } from "system";

printSystemLog("业务前缀：0 start");
printSystemLog(["业务前缀：1 条件数量", value].join(" - "));
```

日志中心：

```text
https://tetris.qunhequnhe.com/log
```

历史资料中的常见服务：

| 场景 | service |
| --- | --- |
| 台面 | `service.dcs-countertop` |
| 设计后处理 | `service.dcs-post-process-task` |
| 输出 JSON | `service.superoutputtask` |
| 订单 | `service.dcs-order-design` |

日志至少覆盖四层：

1. 入口是否触发；
2. 条件是否命中；
3. 商品或结果对象是否创建；
4. 返回数组或变更模型数量。

### 6.3 debugger

历史 CF 资料给出的思路：

1. 获取脚本模块和唯一标识；
2. 在工具中真实触发一次；
3. 在监控平台按唯一标识查找用例；
4. 提取实际入参；
5. 将入参放回编辑器测试用例复现；
6. 对照变量赋值结果定位问题。

## 7. 已验证案例带来的关键经验

### 7.1 业务目标

台面存在台下盆水槽孔时，根据台面厚度生成对应的零价虚拟商品；台上盆、台中盆、无水槽孔和其他厚度不生成。

### 7.2 最终成功链路

最终使用台面业务域：

```text
Fee 自定义对象
  + generateFee(countertop: Countertop): Fee[]
  + ProdCategory.SINK
  + MoldingType.BOTTOM
```

核心判断：

```ts
let sinkBottomHoles = countertop.moldingArea.holes.filter(hole =>
    hole.isProdCategory(ProdCategory.SINK) &&
    hole.isMoldingType(MoldingType.BOTTOM)
);
```

按厚度选择商品，并通过 `Fee.newInstance` 返回：

```ts
let fee = Fee.newInstance({
    price: 0,
    brandgood: targetBrandgood,
    length: 0
});
```

公开资料必须使用 `BG_ID_FOR_20MM`、`BG_ID_FOR_12MM` 等占位符，不保存企业专属商品 ID。

### 7.3 为什么早期方案失败

早期曾尝试在台面脚本中直接使用设计后处理域的 `VirtualModel/addChildren`。由于当前台面域没有暴露 `VirtualModel`，编辑器报“未找到定义”。

错误推断是：

```text
VirtualModel 在台面域不可用
  -> 设计中不能生成虚拟五金
```

正确结论是：

```text
台面域不能直接使用设计后处理对象
  != 整个设计中不支持虚拟商品

台面工艺费/虚拟商品应优先复用同域 Fee + generateFee 链路
```

这条经验适用于其他需求：先寻找同业务域成功案例，不要跨域搬运对象，也不要把“某个类型不可见”扩大成平台能力结论。

### 7.4 日志如何纠正判断

真实日志曾出现：

```text
全部洞数量 - 1
水槽洞数量 - 1
台下盆洞数量 - 0
返回 Fee 数量 - 0
```

设计事实确认该水槽为台上盆，因此“不生成”是正确结果。不要为了得到预期输出而删除正确的安装方式条件。

## 8. 推荐开发流程

1. 明确业务目标和最终可观察结果；
2. 确认业务域、系统入口、固定入参和返回值；
3. 在“可用类型/脚本”中核对依赖；
4. 设计并发布必要的自定义对象；
5. 用最小脚本验证 import 和返回类型；
6. 加入带前缀、序号的日志；
7. 逐步加入业务条件和商品构建；
8. 用编辑器测试用例验证局部逻辑；
9. 发布依赖和入口；
10. 在真实设计或订单流程中触发；
11. 检查业务服务日志和最终 JSON；
12. 记录正向、反向场景和已知限制。

## 9. 排错顺序

遇到问题时依次检查：

1. **权限**：账号是否具备后台维护和前台生效权限；
2. **业务域**：是否选错业务模块；
3. **入口**：唯一标识和函数签名是否正确；
4. **依赖**：对象、子脚本是否发布；
5. **版本**：是否存在当前发布版本；
6. **配置**：索引、业务线、商品和授权是否完整；
7. **语法**：是否使用当前引擎不支持的写法；
8. **触发**：入口日志是否出现；
9. **条件**：过滤数量和关键字段是否符合设计事实；
10. **输出**：返回结果是否进入最终 JSON 或订单链路。

## 10. 权限与入口备注

历史 CF 资料记录过以下权限点：

| 权限点 | 历史用途 |
| --- | --- |
| `11284` | 前台工艺生效 |
| `11285` | 商家后台管理和测试 |

不同业务线和当前权限系统可能已调整。入口通常位于：

```text
商家后台
  -> 厨卫定制 / 全屋定制 / 门窗定制
  -> 高级配置
  -> 工艺编辑器或工艺规则通用配置
```

权限和菜单属于易变信息，实际操作前应以当前商家后台和管理员配置为准。
