# 可共享 Skill 制作

## 划分标准

按稳定能力或业务领域拆分：

```text
什么时候触发
由哪个系统入口调用
输入与输出是什么
依赖哪些工具或对象
如何验证
```

不要按作者拆 Skill，也不要每条很小的脚本单独建 Skill。

## 命名

- Skill 目录：小写英文、数字和连字符；
- frontmatter `name`：与目录名一致；
- 普通参考文件可采用 `english-slug-中文说明.md`；
- `SKILL.md` 必须保留固定名称。

## SKILL.md

```markdown
---
name: example-skill
description: 说明能力、具体触发场景、常见术语和错误关键词。
---

# 标题

## 核心流程

1. ...

## 参考资料

- 需要某类细节时读取 [reference-中文.md](references/reference-中文.md)。
```

frontmatter 只放必要标准字段。`description` 必须同时说明“做什么”和“何时触发”。

## 渐进加载

- `SKILL.md` 只保留决策流程和核心规则；
- 详细案例、API、排错表放 `references/`；
- 可重复执行的确定性逻辑放 `scripts/`；
- 输出模板和资源放 `assets/`。

不要在多个 AI 工具目录复制完整正文。维护一份标准 Skill，通过安装或导入适配不同工具。

## 成功经验要求

共享案例至少包含：

- 业务目标；
- 入口与触发时机；
- 依赖对象和配置；
- 完整可运行内容；
- 正向与反向验证；
- 日志或输出证据；
- 失败过程与修正原因；
- 已知限制；
- 作者和更新时间；
- 脱敏说明。

## 校验

```bash
python scripts/validate-skill-校验Skill.py path/to/skill
```

若目标工具提供官方校验器，再运行对应校验器。

## 评审重点

- 是否把推断写成事实；
- 是否跨执行域误用对象和函数；
- 是否有真实验证；
- 是否重复已有 Skill；
- 是否包含敏感信息；
- 是否能被其他人独立复现。
